# Architecture: RunDeck Agent Mission Control MVP

## 1. Overview

RunDeck is a local-first Windows desktop control surface for launching, supervising, isolating, and reviewing CLI-based AI agent sessions across local and WSL repositories. The MVP uses a desktop monolith architecture: a Tauri v2 shell with React/TypeScript UI, Rust backend commands, SQLite state, file-backed logs, and adapter boundaries for PTY/session execution, WSL runtime execution, git/worktree operations, warning detection, and log redaction.

The system has no SaaS backend, no multi-user authentication, and no autonomous merge path in v1. All workflows remain operator-driven, with worktree isolation and a human review gate as core safety boundaries.

## 3. Tech Stack

| Layer | Technology | Version | Notes |
|-------|------------|---------|-------|
| Desktop Shell | Tauri | 2.x | Windows-first app shell with Rust backend commands |
| Backend Runtime | Rust | Stable, pinned in `rust-toolchain.toml` | Tauri command handlers, services, adapters |
| Frontend Runtime | Node.js | Active LTS, pinned in `.nvmrc` or Volta | Build tooling only |
| Frontend Framework | React | 18+ | TypeScript UI |
| Language | TypeScript | 5.x | Strict mode required |
| Bundler | Vite | 5+ | Tauri-compatible React build |
| Terminal UI | xterm.js | 5.x | Live terminal panes |
| Database | SQLite | 3.44+ | Local structured state |
| Rust SQLite Access | sqlx or rusqlite | Chosen during setup | Must support migrations and typed access |
| YAML Parsing | serde_yaml | 0.9+ | Profile parsing in backend |
| Validation | Zod plus Rust validation | Zod 3+ | UI forms and backend command validation |
| Styling | CSS Modules or Tailwind CSS | Tailwind 3.4+ if selected | Must preserve dense desktop operational UI |
| Testing | Vitest, React Testing Library, Rust tests | Current stable | Unit and component tests |
| Browser Verification | Playwright or dev-browser skill | Current stable | Required for UI stories |
| Containerization | None for MVP runtime | N/A | Local desktop app, not Docker-first |
| Data Location | `%APPDATA%/RunDeck` | N/A | DB, logs, profiles; worktree root configurable |

## 7. Error Handling Strategy

All Tauri command failures return the common error envelope:

```json
{
  "error": {
    "code": "TASK_NOT_FOUND",
    "message": "Task with ID {id} does not exist",
    "details": {},
    "request_id": "req_...",
    "retryable": false
  }
}
```

| Code | When Used |
|------|-----------|
| `VALIDATION_ERROR` | Request payload or form data fails validation |
| `DATABASE_UNAVAILABLE` | SQLite cannot be opened or migrated |
| `APP_DATA_DIR_UNAVAILABLE` | `%APPDATA%/RunDeck` cannot be created/read |
| `WORKSPACE_NOT_FOUND` | Workspace id does not exist |
| `WORKSPACE_ARCHIVED` | Operation requires active workspace |
| `WORKSPACE_SLUG_CONFLICT` | Workspace slug already exists |
| `REPO_NOT_FOUND` | Repo id does not exist |
| `REPO_ARCHIVED` | Operation requires active repo |
| `REPO_PATH_NOT_FOUND` | Repo path does not exist in runtime context |
| `REPO_NOT_GIT` | Repo path is not a git repository |
| `WSL_UNAVAILABLE` | WSL or requested distro is unavailable |
| `PROFILE_NOT_FOUND` | Profile id does not exist |
| `PROFILE_INVALID` | Profile exists but cannot be used |
| `PROFILE_PARSE_ERROR` | YAML profile parse failed |
| `TASK_NOT_FOUND` | Task id does not exist |
| `TASK_STATUS_INVALID` | Invalid task status transition |
| `TASK_LAUNCH_FIELDS_MISSING` | Required launch fields are missing |
| `TASK_ACTIVE_SESSION_EXISTS` | v1 task already has an active session |
| `DIRTY_REPO` | Dirty base repo blocks worktree creation |
| `WORKTREE_CREATE_FAILED` | Branch/worktree creation failed |
| `WORKTREE_MISSING` | Stored worktree path no longer exists |
| `WORKTREE_DELETE_FAILED` | Manual cleanup failed |
| `SESSION_NOT_FOUND` | Session id does not exist |
| `SESSION_NOT_RUNNING` | Stop requested for non-running session |
| `SESSION_RUNNING` | Archive requested for running session |
| `SESSION_LAUNCH_FAILED` | PTY/process launch failed |
| `SESSION_STOP_FAILED` | Process termination failed |
| `GIT_COMMAND_FAILED` | Git command failed |
| `DIFF_UNAVAILABLE` | Diff cannot be generated |
| `REPO_COMMAND_MISSING` | Test/build command is not configured |
| `VALIDATION_LAUNCH_FAILED` | Test/build process could not start |
| `LOG_NOT_FOUND` | Log file missing |
| `LOG_READ_FAILED` | Log file read/search failed |
| `INTERNAL_ERROR` | Unexpected error |

**Logging Rules:**
- Every command receives a `request_id`.
- Backend logs include `request_id`, command name, related ids, and stack/context.
- Logs must never include raw secrets after redaction.
- Error details may include paths and git stderr, but must not include token values, bearer values, or API keys.
- Warning engine failures are logged and surfaced as warnings when practical, but must not crash terminal streaming.

## 8. Security Considerations

**Authentication:**
- No app-level authentication in v1.
- RunDeck is protected by the Windows user account and local filesystem permissions.
- No multi-user auth, RBAC, cloud auth, or app PIN in v1.

**Secrets Management:**
- RunDeck does not store provider API keys as structured secrets in v1.
- Agent CLIs inherit credentials from the operator's environment or existing CLI configuration.
- `.env` files must never be generated or committed by RunDeck.
- Raw unredacted logs are not retained by default.
- Required local paths:
  - `RUNDECK_APP_DATA_DIR` optional override for `%APPDATA%/RunDeck`
  - Profile and log directories under app data unless configured otherwise

**Input Validation:**
- Frontend validates forms with TypeScript/Zod-compatible schemas.
- Backend revalidates all Tauri command inputs.
- YAML profiles are parsed with structured YAML parser, not string splitting.
- Git, WSL, and process execution must avoid raw shell interpolation where practical.
- Launch templates are rendered from explicit variables and audited before execution.
- Repo paths, worktree paths, and log paths are canonicalized before filesystem operations.

**Command Execution:**
- All runtime execution goes through runtime adapters.
- WSL repos must execute git/session/validation commands in WSL, never silently in Windows shell.
- Risky command detection is warning-only in v1 and never automatically kills sessions.
- Required risky patterns must be active by default.

**Log Redaction:**
- Redaction occurs before default saved logs are written and before logs are displayed.
- Required patterns include `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GITHUB_TOKEN`, `AWS_SECRET_ACCESS_KEY`, `AWS_SESSION_TOKEN`, `password=`, and `Authorization: Bearer`.
- Possible secret detection creates a warning.
- Redaction should prefer false positives over leaking obvious secrets.

**Filesystem Safety:**
- Manual `delete_task_worktree` requires the task to be Done or Archived and requires exact path confirmation.
- Deletion must verify the resolved path is under the configured worktree root before removal.
- The app must never delete the base repo path.

**CORS Policy:**
- Not applicable for the v1 local Tauri command API.
- If a local HTTP bridge is introduced later, bind to localhost only and deny all remote origins by default.

## 9. Integration Points

| System | Purpose | Auth Method | Rate Limits |
|--------|---------|-------------|-------------|
| Local Git CLI | Dirty checks, branch creation, worktree creation, status, diff | OS user permissions | Local process limits |
| WSL / `wsl.exe` | Run WSL repo commands and sessions | Windows user WSL access | Local process limits |
| Agent CLIs | Codex, Claude, Gemini, shell, future local agents | Existing CLI/environment auth outside RunDeck | Provider-specific, outside RunDeck |
| Local Filesystem | SQLite DB, YAML profiles, logs, worktrees | OS user permissions | Disk capacity |
| SQLite | Structured app state | Local file access | Local process limits |
| PTY Backend Adapter | Interactive terminal execution | OS process permissions | Local process limits |

RunDeck has no direct external SaaS API dependency in v1. Agent CLIs may call provider APIs, but that network activity is owned by the CLI process and not by RunDeck's application API.

## 10. Non-Functional Requirements

| Requirement | Target | Measurement |
|-------------|--------|-------------|
| Startup Restore | < 2 seconds after app shell load for normal local DB | Restore selected workspace, active tasks, sessions, warnings |
| Command Response Time | < 200ms p95 for standard SQLite CRUD | Workspace, repo, task, warning commands |
| Terminal Stream Latency | < 250ms p95 under four active sessions | Output arrival to pane render |
| Concurrent Sessions | At least 4 active sessions | Usable at 1280x720 |
| WSL Support | At least 1 WSL session in Phase 0 | Launch, stream, stop, log persist |
| Dirty Repo Block Reliability | 100% before coding worktree creation | Dirty state blocks branch/worktree creation |
| Log Persistence | 100% of launched sessions produce redacted log file when process starts | Log exists after session exit/restart |
| App Restart Reconciliation | No stale `running` sessions after restart | Missing processes become `stopped` or `unknown` |
| Data Retention | Indefinite local retention until user archives/deletes artifacts | Historical sessions/logs remain after restart |
| Worktree Cleanup | Manual only for Done/Archived tasks | No surprise deletion |
| UI Minimum Size | 1280x720 | No overlapping terminal controls or task card essentials |
| Token Unavailable Handling | Must never block workflows | Unknown token state still allows launch/review |
| Redaction | Common secret patterns redacted before display/storage | Required patterns covered by tests |
| Accessibility | Keyboard reachable controls and visible focus | UI verification pass |
| Availability | Local app available when OS, filesystem, and configured runtimes are available | No cloud dependency |

## 11. Open Technical Questions

| Question | Current Decision / Required Follow-Up |
|----------|---------------------------------------|
| Which PTY backend should be selected? | Defer final choice. Define `SessionBackend` abstraction and validate Rust PTY vs `node-pty` sidecar in Phase 0. |
| How should PTY streams handle encoding, ANSI control, resize, and backpressure? | Phase 0 must document behavior and failure modes. |
| Which SQLite Rust library should be used? | Choose during scaffold: `sqlx` for compile-time query checks or `rusqlite` for simpler local embedded use. |
| Should validation sessions be visually distinct from agent sessions in terminal grid? | Architecture supports `launch_kind=validation`; UI treatment can be finalized in implementation. |
| What log rotation limit should v1 use? | Not fixed. Must avoid freezing UI with large logs; implement chunked reads now and decide rotation later. |
| Should profile launch templates support advanced templating? | Keep v1 to explicit variable substitution only. |
| How should branch/worktree collision suffixes be generated? | Recommended: append short task id suffix, such as `task/add-priority-filter-a1b2c3`. |
| How should deleted or moved worktrees be reconciled? | Detect on diff/validation/refresh and set `worktree_state=missing`. |
| Should raw unredacted logs ever be optionally retained? | v1 default is no raw retention. Optional raw retention is deferred. |
| How much log search is required? | Simple in-file search over chunked redacted logs for MVP; SQLite FTS deferred. |
| How will future remote Linux support fit? | Keep runtime and session adapters local-first but interface-driven so a remote adapter can be added later. |
